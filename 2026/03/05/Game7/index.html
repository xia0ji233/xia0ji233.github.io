由于该FPS游戏版本变动较大，且 CVE-2025-45737 已经公开，遂公开之前分析。

上来先开驱动。

R0分析

DriverEntry逻辑

结论

DriverEntry按顺序执行以下逻辑

  1. 获取MmGetVirtualForPhysical地址特征码判断反hook反模拟器
  2. MmGetSystemRoutineAddress获取了一系列API
  3. 获取ntoskrnl.exe地址并解析
  4. 读取KeServiceDescriptorTableShadow+0x10获取SSDTshadow的size
  5. 读取IDT获得0xE号缺页异常处理例程地址并进行一定的判断
  6. 读取一些内核标志位，并根据出现的异常情况记录日志
  7. 注册句柄回调，创建进程线程回调，加载模块回调和minifilter回调并建立minifilter端口通信
  8. 创建四个线程，DriverEntry逻辑结束

总结

线程2总体做了这几件事

  • 获取处理器核心个数
  • 判断是否已与游戏完成通信，若没有则直接退出。
  • 根据处理器核心个数创建线程，线程会循环读取 Context 中的 vector 字段并使用中断方式尝试读取指定区域的内存并写入分配的内存中，读取内存字节数被统计计入 Context 的 Value 字段用于日志输出。
  • 若全局变量中的 vector 不为空，则读取并存入 Context 的 vector。

更深层次的利用

可以发现，目前实现的一些功能都有一定的限制，无法直接做到更深层次的利用，比如能够任意的读写内核数据，但是内核中有个标志位可以临时将驱动强制签名关闭，那样便可以达到加载无签名驱动的目的，而关键的判断则是 ci.dll 中的某个全局变量

当 g_CiOptions==0 时，可以加载任何签名，g_CiOptions==6 时，驱动必须强制签名，g_CiOptions==8 时，可以使用测试签名，所以这里在应用层用 NtQuerySystemInformation 获取 ci.dll 模块的基址，再根据偏移将该值写入 0 达到绕过驱动强制签名的限制。理论上，实时读写 system32 下对应的 ci.dll 然后下载 pdb 解析可以实现通杀。
